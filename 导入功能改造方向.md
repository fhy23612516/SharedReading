# 导入功能改造方向

本文记录当前导入书籍功能的已知问题和后续改造方向。

## 当前问题

1. 文件编码不稳：当前前端按 `utf-8` 读取 `.txt`，遇到 GBK / GB18030 中文 TXT 可能乱码。
2. 大小限制不一致：后端 `parseBody` 原始请求体限制约 1MB，但导入文本逻辑允许约 50 万字符；中文 JSON 传输时可能超过 1MB，导致还没进入业务校验就失败。
3. 章节识别缺失：当前只按空行拆段，没有识别“第 X 章 / 第 X 回 / Chapter”等章节结构。
4. 导入前缺少预览：用户不知道系统会拆成多少段、多少字、是否乱码、是否过长。
5. 错误反馈偏粗：失败后只给通用 toast，没有明确区分编码、大小、网络、未登录、格式等原因。
6. 长文本体验弱：没有读取进度、导入中状态、重复点击保护和取消导入。
7. 内容治理缺失：当前只做长度限制，没有敏感词、重复内容、异常文本或垃圾内容检查。

## 优先改造顺序

### P0：让导入稳定可用

1. 前端读取文件时优先尝试 `utf-8`，乱码明显时提示用户切换编码。
2. 增加编码选项：`UTF-8`、`GBK/GB18030`。
3. 前端在提交前检查原始字符数和估算请求体大小。
4. 后端把导入接口请求体限制提高到与业务上限一致，或把业务上限降到真实可提交范围。
5. 导入按钮增加 loading 状态，提交中禁止重复点击。
6. 明确错误文案：未登录、正文太短、正文太长、请求体过大、网络失败、JSON 无效。

### P1：让导入结果可控

1. 增加导入预览页或预览区域。
2. 展示识别出的标题、作者、字数、段落数、章节数。
3. 支持用户手动调整标题、作者、标签和简介。
4. 增加章节识别规则：
   - `第十二章 标题`
   - `第十二回 标题`
   - `Chapter 12`
   - 空行分段兜底
5. 导入后进入“导入成功”页，提供“创建房间”“加入书架”“继续导入”。

### P2：让导入适合长期使用

1. 支持 EPUB 解析。
2. 支持按章节存储，避免整本书作为一个超长正文。
3. 支持书籍详情页和目录页。
4. 支持导入历史、重新编辑、删除导入书籍。
5. 支持同名书籍去重提醒。
6. 支持导入后的封面生成或用户自定义封面。

## 推荐技术方案

1. 前端读取文件为 `ArrayBuffer`，再用 `TextDecoder` 按用户选择编码解码。
2. 前端先本地完成解析和预览，再提交结构化结果。
3. 后端导入接口拆成两层：
   - `POST /api/books/import/preview`：可选，返回解析结果。
   - `POST /api/books/import`：保存确认后的书籍。
4. MySQL 长期方案应从 `books.body_json` 进一步拆出章节表，例如 `book_chapters`。
5. 如果后续做小程序端导入，优先从云端文件或粘贴文本开始，不先做复杂本地文件解析。

## 下一步建议

先做 P0 和 P1，不急着做 EPUB。当前最大风险是编码乱码和请求体大小不一致，应该优先修。
