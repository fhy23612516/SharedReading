# 双人实时同步阅读产品需求文档

## 1. 产品名称

暂定名：一起等你读

## 2. 产品一句话介绍

一个让两个人一起读同一篇内容的实时共读产品：当一个人读得太快，系统会让他等一等对方，让阅读变成一种陪伴体验。

## 3. 产品定位

本产品不是普通小说阅读器，而是一个“陪伴式阅读”工具。

它的核心不是让用户最快读完内容，而是让两个人保持相近节奏，一边阅读，一边互动。

适合以下场景：

- 情侣一起读小说
- 朋友一起读故事
- 陪玩/陪聊场景下的共同阅读
- 读书搭子一起共读
- 粉丝和陪读员一起读内容
- 两个人一起看剧情、看短篇、看连载

## 4. 核心创意

传统阅读是一个人读。

本产品把阅读变成两个人同步体验。

核心机制：

当两个人阅读进度差距过大时，进度较快的一方会触发“等待”。

等待不是惩罚，而是一种陪伴感设计。

用户会看到类似提示：

> 你读得有点快啦，等 TA 一下。

当较慢的一方追上来后，等待自动解除。

## 5. 产品目标

第一版目标是做出一个可体验的最小版本，验证以下事情：

1. 两个人是否愿意一起读同一篇内容。
2. “读太快要等对方”这个机制是否有趣。
3. 等待过程是否能增强陪伴感。
4. 两个人边读边聊是否自然。
5. 这个模式是否适合陪玩圈、陪聊圈、读书搭子、情侣社交等场景。

## 6. 目标用户

### 6.1 主要用户

#### 情侣用户

两个人想一起读小说、故事、情感内容，希望有一种“同步陪伴”的感觉。

#### 朋友用户

两个朋友想一起看同一篇内容，一边读一边吐槽。

#### 陪玩/陪聊场景用户

老板和陪玩/陪聊人员一起阅读剧情、小说、互动文本，用阅读作为陪伴服务的一部分。

#### 读书搭子

两个人约定一起读内容，通过同步进度保证双方节奏一致。

### 6.2 次要用户

- 小说爱好者
- 剧情互动内容爱好者
- 喜欢社交阅读的人
- 需要陪伴感的人
- 想要新型互动体验的用户

## 7. MVP 范围

### 7.1 第一版必须包含

2. 加入共读房间

## 8. 用户流程

## 8.1 创建房间流程

1. 用户打开产品。
2. 点击“创建共读房间”。
3. 选择一篇阅读内容。
4. 设置等待阈值，也可以使用默认值。
5. 创建房间。
6. 系统生成房间码或邀请链接。
7. 用户分享给另一位用户。
8. 创建者进入等待页面。
9. 另一位用户加入后，双方进入阅读页面。

## 8.2 加入房间流程

1. 用户打开别人分享的链接，或输入房间码。
2. 系统检查房间是否存在。
3. 系统检查房间人数是否已满。
4. 如果房间可加入，则进入房间。
5. 加入成功后，双方开始共读。

## 8.3 共读流程

1. 两个用户进入同一个阅读房间。
2. 双方看到同一篇内容。
3. 双方可以自由向下阅读。
4. 页面实时显示自己和对方的阅读进度。
5. 如果一方读得过快，系统触发等待。
6. 进度快的一方暂时不能继续向下读。
7. 进度慢的一方继续阅读。
8. 当两人进度差距回到合理范围，等待解除。
9. 两人继续阅读。
10. 读完后进入阅读完成页。

## 8.4 聊天流程

1. 用户在阅读房间内打开聊天区。
2. 输入消息。
3. 点击发送。
4. 对方实时收到消息。
5. 聊天记录保留在本次房间中。

## 10. 核心规则

## 10.1 房间规则

1. 一个房间最多 2 人。
2. 创建者默认是房主。
3. 第二个人加入后，房间开始。
4. 房间开始后，不允许第三个人加入。
5. 任意一方退出，另一方需要看到提示。
6. 房间完成后变为已结束状态。
7. 已结束房间不能继续加入。

## 10.2 进度规则

1. 阅读进度使用百分比表示。
2. 进度范围是 0% 到 100%。
3. 用户向下阅读时，进度增加。
4. 用户向上回看时，可以显示当前位置，但总进度不应该异常倒退。
5. 第一版可以用滚动位置计算进度。
6. 进度需要实时同步给对方。
7. 进度更新不需要每一毫秒同步，保持流畅即可。

## 10.3 等待规则

### 触发条件

当两人阅读进度差距超过房间设置的阈值时，进度更快的一方进入等待状态。

示例：

- 阈值：8%
- A 进度：40%
- B 进度：30%
- 差距：10%
- A 比 B 快
- A 进入等待

### 解除条件

当两人进度差距小于或等于阈值时，等待解除。

示例：

- 阈值：8%
- A 进度：40%
- B 进度：33%
- 差距：7%
- A 解除等待

### 等待期间允许做的事

等待中的用户可以：

1. 看进度
2. 看提示
3. 聊天
4. 回看前文
5. 退出房间

等待中的用户不可以：

1. 继续向下推进阅读进度
2. 跳到后面内容
3. 绕过等待机制

## 10.4 慢速用户规则

读得慢的一方不应该被惩罚。

不要显示羞辱性或压力性文案。

不要写：

> 你太慢了。

> 你拖累了对方。

> 请加快速度。

推荐写：

> 对方正在等你。

> 你们快同步啦。

> 继续读就能一起往下看。

## 11. 等待体验设计

## 11.1 等待不是惩罚

等待机制的感受应该是温柔的、陪伴的、有趣的。

用户应该感觉：

> TA 在等我。

而不是：

> 系统在限制我。

## 11.2 等待文案

进度快的一方看到：

> 你读得有点快啦，等 TA 一下。

> TA 正在追上你。

> 还差 3% 就可以继续啦。

> 可以先聊聊刚才这段。

进度慢的一方看到：

> TA 正在等你。

> 你们快同步啦。

> 慢慢读，不着急。

> 读到这里就能一起继续啦。

## 11.3 等待互动

等待时可以提供快捷互动：

1. 发送“我等你”
2. 发送“慢慢读”
3. 发送“这里好看”
4. 发送“看到这里告诉我”
5. 发送表情
6. 打开聊天区

## 20. 验收标准

## 20.1 创建房间

满足以下条件：

1. 用户可以创建房间。
2. 房间可以生成唯一房间码。
3. 创建者可以进入等待页面。
4. 房间信息保存成功。

## 20.2 加入房间

满足以下条件：

1. 第二个用户可以通过房间码加入。
2. 加入后双方进入同一阅读房间。
3. 第三个用户不能加入。
4. 房间不存在时有明确提示。
5. 房间已满时有明确提示。

## 20.3 阅读同步

满足以下条件：

1. 双方可以看到同一篇内容。
2. 双方可以看到自己进度。
3. 双方可以看到对方进度。
4. 一方滚动阅读后，对方能看到进度变化。
5. 进度显示基本准确。

## 20.4 等待机制

满足以下条件：

1. 当双方进度差超过阈值时，进度快的一方进入等待。
2. 进度慢的一方不被限制。
3. 等待中的用户不能继续向下推进进度。
4. 等待中的用户可以聊天。
5. 当进度差回到阈值内，等待自动解除。
6. 等待开始和结束都有明确提示。

## 20.5 聊天功能

满足以下条件：

### 26.11 弱网快速响应与离线补发

当前版本增加了“本地优先 + 自动补发”的弱网策略，目标是让用户操作先有反馈，再等待网络同步。

1. 阅读进度先在本机页面即时更新，再向后端发送。
2. 聊天消息先进入本地聊天区，并带上 `clientId`，避免弱网重试造成重复消息。
3. 进度和消息请求设置较短超时，超时后不会卡住界面。
4. 失败的进度和消息会写入浏览器本地 `outbox` 队列。
5. 网络恢复、重新进入房间或 SSE 连接恢复后，会自动补发队列。
6. 进度补发使用“最后进度优先”，同一房间同一用户只保留最高进度。
7. 消息补发使用 `clientId` 幂等，服务端发现同一条消息已保存时直接返回旧消息。
8. 后端状态写入改为异步合并落盘，减少每次进度和消息更新时的阻塞。
9. SSE 推送改为局部事件，包括 `progress`、`message`、`presence`，减少全量房间数据传输。

弱网补发流程：

```text
用户操作
  |
  |-- 前端立即更新页面
  |
  |-- 请求后端
        |
        |-- 成功：用服务端返回结果校正本地状态
        |
        |-- 超时/断网：写入本地 outbox
                         |
                         |-- 网络恢复或重新进入房间
                               |
                               |-- 自动补发
                                     |
                                     |-- 服务端保存并通过 SSE 通知双方
```

## 28. 已完成基础实现：账号、反馈、MySQL 与性能

本阶段新增需求以 `后续迭代设计-账号反馈MySQL性能.md` 为准。当前已经完成基础实现，生产服务器启用 MySQL 的操作步骤见 `MySQL迁移与启用.md`。

已实现目标：

1. Web 和小程序使用同一套自有账号系统。
2. 不依赖微信/QQ 登录。
3. 支持 MySQL 替代 `data/store.json`，默认仍使用 JSON。
4. 增加用户反馈功能。
5. 通过 `/api` 直连、轻量接口和本地优先补发提升响应体验。

已完成实施顺序：

1. MySQL 存储层。
2. 统一注册登录。
3. 反馈功能。
4. 响应时间优化。

关键接口：

1. `POST /api/auth/register`
2. `POST /api/auth/login`
3. `POST /api/auth/logout`
4. `GET /api/auth/me`
5. `POST /api/auth/password/reset`
6. `POST /api/auth/recovery-code`
7. `POST /api/feedback`
8. `GET /api/feedback/mine`

账号原则：

1. 账号字段使用 `account`，可先作为用户名，后续兼容手机号或邮箱。
2. 密码只存哈希，不存明文。
3. Web 用 `localStorage` 保存 token。
4. 小程序用本地 storage 保存 token。
5. 写接口优先通过 `Authorization: Bearer <token>` 识别用户，同时短期兼容旧的 `userId` 和 `name`。
6. 注册时生成一次性密码恢复码，只返回给用户，不保存明文。
7. 找回密码使用“账号 + 恢复码 + 新密码”；重置成功后旧恢复码和旧登录会话失效。
8. 登录用户可以重新生成恢复码，新恢复码生成后旧恢复码立即失效。

## 29. 阅读平台基础功能补充（2026-04-29）

本节记录向常见阅读产品靠拢后的基础能力。当前仍以“双人一起阅读”为核心，不做复杂书城、付费、推荐算法和版权内容平台。

### 29.1 已实现能力

1. 搜索：支持按标题、作者、简介和标签搜索内置内容与当前账号导入内容。
2. 书架：登录用户可把内容加入书架，并在书架中继续阅读。
3. 浏览记录：登录用户进入房间或上报进度后，会记录最近阅读内容、房间和最大进度。
4. 章节评论：围绕整篇内容发布评论，适合表达整体感受。
5. 段落评论：围绕具体段落发布评论，阅读页会显示对应段评数量。
6. 评论汇总：阅读页右侧显示当前章节或段落下的评论列表。
7. 搜索结果和书架卡片会显示阅读进度、是否已加入书架和评论数量摘要。
8. MySQL 与 JSON 两种存储都支持上述数据。

### 29.2 产品规则

1. 书架、浏览记录、发布评论都需要登录。
2. 未登录用户可以查看评论，但不能发布评论、加入书架或保存浏览记录。
3. 搜索未登录时只返回内置内容；登录后额外返回当前账号导入的内容。
4. 段落评论必须绑定合法段落序号。
5. 评论单条最多 300 字。
6. 浏览记录按最后阅读时间倒序展示。
7. 书架按最后加入或更新时间倒序展示。

5. 评论互动：点赞、回复、举报、作者置顶、审核。
6. 阅读资产同步：划线、书签、字体、主题同步到账号。
7. 连续阅读：从上次段落或章节自动继续。
8. 内容运营：推荐位、专题、榜单、活动房间。

## 30. 导入功能改造方向（2026-04-30）

当前导入功能已经完成一轮稳定性增强：

1. Web 导入页支持选择 TXT 编码：`UTF-8`、`GBK / GB18030`。
2. 选择文件后会本地解码并自动填入正文，书名为空时会用文件名回填。
3. 导入前会显示字数、段落数、章节识别数量和请求体大小估算。
4. 章节识别先覆盖“第 X 章”“第 X 节”“第 X 回”“第 X 卷”“第 X 部”“Chapter X”等常见标题。
5. 前端会在提交前拦截正文太短、正文太长和请求体过大的情况。
6. 后端导入接口请求体上限已提高到约 2.5MB，单本文本仍限制约 50 万字符。
7. 导入按钮有提交中状态，避免重复点击。
8. 错误提示已区分未登录、正文太短、正文太长、请求体过大、JSON 异常、请求超时和网络异常。
9. 超过单次导入上限的大书会自动走分章导入。
10. 分章导入会写入 `book_chapters`，创建房间时选择具体章节共读。
11. 分章导入支持章节标题识别；识别不到章节时按固定长度切片。
12. 当前大书总文本上限约 500 万字符，单章上限约 12 万字符，最多 1000 章。

仍未完成的长期能力：

1. EPUB/PDF 等复杂格式导入。
2. 正式书籍详情页和完整目录页。
3. 导入历史、重新编辑、删除导入书籍。
4. 同名书籍去重提醒。
5. 内容治理、敏感词、反垃圾和版权风险提示。
6. 整本书连续阅读和跨章节进度同步。

详细方案见 `导入功能改造方向.md`。

## 31. 大书阅读性能与手机进度修正（2026-05-01）

本节记录 `v0.8.1` 对大书和移动端共读体验的修正。

### 31.1 已实现规则

1. 首页内容列表、搜索结果、我的导入、书架、浏览记录和导入成功响应都只返回书籍元数据，不返回完整正文。
2. 只有进入房间后，房间详情才返回当前阅读内容正文。
3. 分章书籍创建房间时仍以具体章节为阅读单位，避免整本大书一次性进入房间。
4. 在线心跳只返回在线状态和等待状态，不返回完整房间和正文。
5. 阅读进度计算同时支持桌面端内部阅读容器滚动和手机端整页滚动。
6. 手机端添加书签时使用当前真实阅读位置，点击书签也能跳回对应位置。

### 31.2 解决的问题

1. 大书导入后，另一台设备进入房间或页面初始化时不再因为列表接口携带大量正文而明显卡顿。
2. 手机浏览器在小屏布局下滚动的是页面本身时，也能上报阅读进度。
3. 心跳请求不再反复传输正文，弱网环境下更稳定。

### 31.3 仍需注意

1. 当前房间正文仍是一次性加载当前章节，超长章节仍可能造成手机渲染压力。
2. 后续如果导入内容以几十万字单章存在，应继续拆成更小章节或做段落懒加载。
3. 旧的超长单本非分章导入内容，建议重新按分章方式导入。
